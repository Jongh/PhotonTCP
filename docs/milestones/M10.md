# M10 — cv2 QR 디코드 견고화

## 목표

실 광학 링크에서 카메라가 잡은 QR 프레임의 디코드 성공률을 끌어올린다. 현재 cv2
`QRCodeDetector`가 특정 콘텐츠의 유효한 QR을 어느 배율로도 디코드하지 못하는
콘텐츠 의존 블라인드스폿이 존재하는데, 이 마일스톤이 끝나면 `decode_frame`이
전처리 캐스케이드(이진화·샤프닝·업스케일)와 대체 detector 폴백으로 그 블라인드스폿
대부분을 회복하고, 결정적 코퍼스 하니스로 디코드율을 회귀 검증할 수 있게 된다.
높아진 수신율은 v1.0 수동 사인오프(셀프체크 ≥80%)의 통과 여유를 만들고, 디코드가
견고해진 토대 위에서 M9가 보류한 다중 바이트 nonce 경로를 안전하게 재검증한다.

## 배경

M9 리뷰(`docs/reports/M9-review.md`)의 후속 메모에서 곧장 나온다:

- **권장 2 (미해결)**: 특정 v8 QR을 cv2 `QRCodeDetector`가 어느 배율로도 디코드 못
  하는 사례가 확인됐다. 실 링크에선 ARQ가 새 nonce→새 이미지로 재전송해 회복하지만
  **수신율을 깎고 지연을 늘린다**. 대체 detector·전처리(이진화/샤프닝)·EC 상향 등
  디코드 견고화가 실 수신율과 v1.0 통과율에 직결되는 후속 과제로 명시됐다.
- **권장 1 (보류된 다중 바이트 nonce)**: M9-T03에서 2바이트 nonce가 한 세션 프레임을
  바로 이 블라인드스폿에 빠뜨려 회귀를 냈고(업스케일 폴백도 무효), `_NONCE_BYTES=1` +
  윈도우 dedup으로 우회했다. 리뷰는 "**디코드 견고화 후** nonce 폭을 QR 콘텐츠에서
  분리"하는 보강을 후속 검토하라고 남겼다 — 본 마일스톤이 그 선행 조건을 만든다.
- **현재 코덱**: `qr/encode.py`는 EC 기본 `"m"`(~15%), `qr/decode.py`는 단일
  `cv2.QRCodeDetector.detectAndDecode`에 그레이스케일 정규화만 적용한다. 전처리·폴백·
  EC 조정 여지가 모두 열려 있다.

이번 사이클은 디코드 견고화에 집중하고, 실 하드웨어 사인오프(v1.0 게이트)는 M9대로
수동 절차로 둔다.

## 태스크 목록

- M10-T01: `decode_frame` 전처리 캐스케이드(이진화·샤프닝·업스케일) 추가
- M10-T02: 대체 QR detector(WeChat) 폴백 + 우아한 미가용 처리 (deps: M10-T01)
- M10-T03: 광학 경로 EC 레벨 견고화 — `OpticalChannel` 인코드 EC 설정 가능화
- M10-T04: 디코드 견고성 코퍼스 + 벤치 스크립트 + 임계 테스트 (deps: M10-T01, M10-T02)
- M10-T05: 다중 바이트 nonce 재검증 (견고해진 디코드 기준) (deps: M10-T02, M10-T03, M10-T04)
- M10-T06: 테스트 + 문서 (deps: M10-T01, M10-T02, M10-T03)

## 태스크 상세

### M10-T01: `decode_frame` 전처리 캐스케이드

- **변경 파일**: `photontcp/qr/decode.py`
- 현재 `decode_frame`은 그레이스케일 정규화 후 `detector.detectAndDecode`를 1회만
  호출한다. 입력 이미지에 대해 **결정적 변형 캐스케이드**를 순서대로 시도하고 첫
  성공에서 단락하도록 바꾼다. 변형 순서(저비용→고비용):
  1. 원본 그레이스케일(현행 경로 — 정상 프레임의 비용·동작 불변).
  2. Otsu 이진화(`cv2.threshold(..., THRESH_BINARY|THRESH_OTSU)`).
  3. 언샤프/샤프닝 커널(`cv2.filter2D`) 후 이진화 — 흐릿한 카메라 캡처 대응.
  4. 업스케일 2.0×, 3.0×(`cv2.resize`, `INTER_NEAREST` 또는 `INTER_CUBIC`) — M9에서
     단독으론 무효였던 업스케일을 전처리와 **조합**해 재시도.
- 변형 생성은 내부 헬퍼(`_decode_variants(image) -> Iterator[np.ndarray]` 같은)로
  분리해 T02의 폴백 detector도 같은 변형들을 재사용할 수 있게 한다.
- 계약 불변: 모든 변형이 실패하면 여전히 `None` 반환(절대 raise 안 함). 모든
  `cv2.error`는 변형 단위로 잡아 다음 변형으로 넘어간다(한 변형 실패가 전체를 죽이지
  않게). 성공 시 base64 디코드 경로는 기존과 동일.
- 핫패스 비용 주의: 정상 프레임은 변형 1에서 즉시 성공해 추가 비용 0이어야 한다.
  변형 2~4는 **첫 시도 실패 시에만** 평가(지연 생성).

### M10-T02: 대체 QR detector(WeChat) 폴백 (deps: M10-T01)

- **변경 파일**: `photontcp/qr/decode.py`
- 기본 `cv2.QRCodeDetector`가 모든 전처리 변형에서 실패하면 **대체 detector**로 한 번
  더 시도한다. 1순위 후보는 opencv-contrib의 `cv2.wechat_qrcode_WeChatQRCode`(별도
  모델 파일 없이 생성 가능한 경우) — 콘텐츠 의존 블라인드스폿에 강한 것으로 알려짐.
- **우아한 미가용 처리**: `cv2.wechat_qrcode`가 없거나(코어 opencv-python) 생성이
  실패하면(모델 파일 필요) 조용히 건너뛴다 — import/생성은 모듈 로드가 아니라 최초
  사용 시 `try/except (AttributeError, cv2.error)`로 감싸고, 사용 가능 여부를 모듈
  수준 플래그/지연 캐시로 1회만 판정. detector도 thread-local 재사용(현 패턴과 일관).
- 대체 detector 미가용 환경에서는 동작이 T01과 동일(폴백 없이 None) — 즉 이 태스크는
  **추가 회복만** 제공하고 기존 환경을 깨지 않는다.
- 조사 불확실성: WeChat detector가 모델 파일 없이 동작하는지는 설치 버전에 의존 →
  미동작 시 **대체 경로**로 `cv2.QRCodeDetectorAruco`(OpenCV ≥4.7) 시도, 그것도 없으면
  폴백 생략. 어느 경우든 코어 동작은 T01 캐스케이드로 보장된다.

### M10-T03: 광학 경로 EC 레벨 견고화

- **변경 파일**: `photontcp/optical/channel.py`
- `encode_frame`은 이미 `error` 파라미터(기본 `"m"`)를 받는다. `OpticalChannel`이
  프레임을 인코드할 때 EC 레벨을 **설정 가능**하게 하고(생성자 인자 예: `error="q"`),
  실 링크 기본값을 한 단계 올리는 것을 검토한다(`"q"` ≈ ~25% 복원). 높은 EC는 모듈 수
  ↑(용량 ↓)지만 카메라 캡처 견고성 ↑.
- 기존 호출부 호환: 인자에 기본값을 주어 시그니처 변경이 backward-compatible이게 한다.
  채널이 `encode_frame(..., error=self._error)`로 전달하도록 배선.
- 엣지: EC 상향으로 동일 페이로드가 더 큰 QR 버전이 되어 `QRCapacityError` 임계가
  내려갈 수 있다 — 현 패킷 크기에서 단일 심볼에 들어가는지 확인하고, 들어가지 않으면
  기본값은 `"m"` 유지하되 인자로 상향만 열어둔다(테스트로 경계 확인).
- T01/T02와 파일 분리 → 병렬 구현 안전.

### M10-T04: 디코드 견고성 코퍼스 + 벤치 + 임계 테스트 (deps: M10-T01, M10-T02)

- **변경 파일**: `tests/test_qr_robustness.py`(신규), `examples/qr_decode_bench.py`(신규)
- **결정적 코퍼스**: 다양한 페이로드(길이·바이트 분포 스펙트럼 + M9에서 블라인드스폿을
  유발한 것으로 확인된 콘텐츠 패턴 포함)를 시드 고정으로 생성 → `encode_frame`으로
  이미지화 → `decode_frame`으로 왕복. 카메라 노이즈를 흉내 내는 결정적 열화(가벼운
  블러·다운스케일 후 업스케일)를 옵션으로 적용해 "캡처 유사" 케이스도 측정.
- **벤치 스크립트**(`examples/qr_decode_bench.py`): 코퍼스 디코드율을 사람이 읽는
  형태로 출력(전처리 캐스케이드/폴백 on·off 비교 포함). 하드웨어 불필요, 순수 인메모리.
- **임계 테스트**(`tests/test_qr_robustness.py`): 코퍼스 왕복 성공률이 기준 임계
  이상임을 단언(예: 깨끗한 코퍼스 100%, 열화 코퍼스 ≥ 합의된 하한). T01/T02 적용 후의
  실제 수치로 임계를 보수적으로 고정해 회귀 가드로 만든다. **블라인드스폿 콘텐츠가
  이제 디코드됨**을 명시적 케이스로 단언.
- 파일 신규라 T03과 병렬 안전이나, 임계 수치는 T01·T02 결과에 의존 → deps 표기.

### M10-T05: 다중 바이트 nonce 재검증 (deps: M10-T02, M10-T03, M10-T04)

- **변경 파일**: `photontcp/optical/channel.py`, `tests/test_optical_pacing.py`
  (또는 nonce 관련 테스트 파일)
- 디코드가 견고해진 지금, M9가 보류한 **다중 바이트 nonce**(권장 1)를 재검증한다.
  코퍼스/벤치(T04)로 2바이트 nonce가 더 이상 블라인드스폿 회귀를 내지 않는지 측정.
- 판단 분기:
  - 회귀가 사라졌으면 `_NONCE_BYTES`를 늘려 윈도우 dedup의 거짓-dedup 코너(권장 1의
    이론적 래핑)를 수학적으로도 제거하고, 회귀 테스트 추가.
  - 여전히 특정 콘텐츠에서 불안정하면 nonce 폭을 **그대로 두되**, 대안으로 nonce를
    base64 페이로드(QR 콘텐츠) 안이 아니라 별도 채널 헤더로 분리하는 방향을 문서화하고
    실제 폭 확대는 다음 사이클로 미룬다(이 경우 코드 변경 최소).
- 어느 분기든 결과를 impl 보고서에 근거 수치와 함께 기록한다(설계 결정의 추적성).
- T03과 같은 `channel.py`를 건드리므로 deps로 직렬화.

## 완료 기준

1. 정상(깨끗한) QR 이미지를 `decode_frame`에 주면 전처리 변형 1에서 즉시 디코드되어
   기존과 동일한 바이트를 반환하고, 추가 변형 비용이 발생하지 않는다(핫패스 불변).
2. M9에서 cv2 `QRCodeDetector`가 디코드하지 못하던 블라인드스폿 콘텐츠를
   `encode_frame`→`decode_frame` 왕복하면, 전처리 캐스케이드 또는 대체 detector 폴백을
   통해 **원본 바이트가 복원**된다(해당 케이스를 단언하는 테스트 통과).
3. 대체 detector가 없는 환경(코어 opencv-python, 모델 파일 부재)에서도 `decode_frame`이
   예외 없이 동작하고, 디코드 불가 프레임은 여전히 `None`을 반환한다.
4. `OpticalChannel`을 `error="q"`로 생성하면 그 EC 레벨로 프레임이 인코드되며, 인자
   없이 생성하면 기존 동작과 호환된다.
5. `python examples/qr_decode_bench.py` 실행 시 캐스케이드/폴백 on·off의 코퍼스 디코드율이
   출력되고, on 쪽이 off 대비 같거나 높다(하드웨어 불필요, exit 0).
6. `python -m pytest -q`가 전부 통과하고(기존 175 + 신규, 회귀 0),
   `tests/test_qr_robustness.py`의 코퍼스 임계 단언이 통과한다.
7. M10-T05의 다중 바이트 nonce 재검증 결과(확대/보류 중 택일)와 그 근거 수치가
   impl 보고서에 기록된다.
