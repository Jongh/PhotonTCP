# M6 — 파일 전송 (단방향 + 완료 핸드셰이크)

## 목표

PhotonTCP 위에 **신뢰성 있는 단방향 파일 전송**을 올린다. 송신측이 파일을 OFFER(이름·크기·SHA-256)
하면 수신측이 ACCEPT하고, 파일을 청크로 잘라 신뢰성 스트림으로 보내며, 마지막에 **앱 레벨 완료
핸드셰이크**(FILE_DONE → 수신측 전체 SHA-256 검증 → FILE_ACK/FILE_NACK)로 무결성을 확인한 뒤에야
송신측이 세션을 종료한다(M4 리뷰의 flush-on-close 권장 해소). 이 마일스톤이 끝나면 **손실 있는 채널 +
가상 클럭에서 한 파일이 손실 없이 전부 전달되고 수신 파일의 SHA-256이 원본과 일치**함이 결정적으로
검증된다 — 로드맵 5단계 "파일 전송".

## 배경

로드맵 5단계. M4에서 채팅(메시지 앱)이 신뢰성 스트림 위에 올라갔고, 파일 전송도 동일하게 스트림 위의
애플리케이션이다. 회고(`docs/reports/retro.md`)에서 **최우선 미반영 후속 = flush-on-close(M4-review)**
로 부각됐고, 본 마일스톤에서 앱 레벨 완료 핸드셰이크로 이를 해소한다.

마일스톤 작성 전 4개 설계 결정을 사용자와 확정했다(모두 반영):

1. **프레이밍 = 단일 스트림 인터리브** — 한 스트림에 타입 구분 길이접두 프레임. 제어는 JSON
   (OFFER/ACCEPT/REJECT/DONE/ACK/NACK), 청크는 바이너리. 채팅 코덱과 같은 길이접두 패턴 + 타입 바이트.
2. **무결성 = 전체 파일 SHA-256** — OFFER에 SHA-256 포함, 수신 완료 후 재조립 파일 해시와 비교. 청크
   단위 무결성은 ARQ/CRC가 이미 보장하므로 전체 해시로 충분(stdlib `hashlib`).
3. **완료/종료 보장 = 앱 레벨 완료 핸드셰이크** — 송신측은 FILE_DONE 후 **FILE_ACK를 받은 뒤에만**
   세션을 종료. 트랜스포트 flush API 불필요, 무결성도 동시 확인. (수신측이 SHA 불일치면 FILE_NACK.)
4. **범위 = 단방향 단일 파일 + 완료확인 + 진행률** — 한 파일을 한 방향으로. 재개(resume offset)·
   양방향·다중 파일은 후속 마일스톤으로 이월.

설계 원칙 유지: 앱은 신뢰성 스트림(M3 ARQ) + 다중화(M4 StreamMux) 위에 얹는 **순수 코덱 + 동기 펌프
드라이버**. 시각은 주입 clock, 결정적. **Session 계층은 수정하지 않는다**(완료 핸드셰이크가 앱 레벨이라
flush API가 불필요 — 채팅처럼 순수 추가). 전용 스트림(`open_stream`)을 써서 채팅과 병행 가능.

## 공유 규약 (모든 태스크 준수)

- **파일 프레임 포맷**: `4바이트 big-endian 길이(이후 바이트 수)` + `1바이트 타입` + `본문`. 길이는
  `1(타입) + len(본문)`. 제어 프레임 본문 = JSON(UTF-8), 청크 프레임 본문 = 원시 바이트.
- **프레임 타입(`FileFrameType` IntEnum)**: `OFFER`, `ACCEPT`, `REJECT`, `CHUNK`, `DONE`, `ACK`, `NACK`.
- **OFFER 본문(JSON)**: `{"name": str, "size": int, "sha256": hex str}`. **DONE/ACK/NACK 본문**: JSON
  (빈 객체 또는 `{"reason": ...}`/`{"ok": bool}`). **CHUNK 본문**: 원시 파일 바이트(스트림이 순서를
  보장하므로 청크 순번 불필요 — 도착 순서대로 이어붙임).
- 스트림이 신뢰성·순서를 보장하므로 코덱은 재전송/순서를 다루지 않는다(순수 프레이밍만).

## 태스크 상세

### M6-T01: 파일 전송 프레임 코덱 + SHA 헬퍼

- **변경 파일**: `photontcp/app/file_codec.py`
- 순수, stdlib(`json`, `struct`, `hashlib`, `enum`, `dataclasses`)만.
- `class FileFrameType(IntEnum)`: OFFER/ACCEPT/REJECT/CHUNK/DONE/ACK/NACK.
- `@dataclass FileFrame`: `type: FileFrameType`, `body: bytes`(원시) — 또는 제어용 헬퍼로 JSON 본문 직렬화/역직렬화 제공.
- `encode_frame(type: FileFrameType, body: bytes = b"") -> bytes`: `struct.pack(">I", 1+len(body)) + bytes([int(type)]) + body`.
- 제어 헬퍼: `encode_control(type, obj: dict) -> bytes`(JSON 본문), 그리고 `decode_control(body) -> dict`.
- `class FileFrameReassembler`: `feed(self, data: bytes) -> list[FileFrame]` — 길이접두로 완전 프레임만 추출(부분 잔류), 타입 바이트 파싱. 비현실적 길이/타입 방어(`MAX_FRAME_BYTES` 등, 예외 또는 스킵 명확히).
- `sha256_hex(data: bytes) -> str`: `hashlib.sha256(data).hexdigest()`.
- 모듈/함수 docstring + 공유 규약 명시.

### M6-T02: `FileSender`/`FileReceiver` 앱 (deps: M6-T01)

- **변경 파일**: `photontcp/app/file.py`, `photontcp/app/__init__.py`(re-export에 `FileSender`/`FileReceiver`/`FileFrameType`/`FileTransferState` 등 추가; 기존 chat/codec re-export 유지)
- M3 ARQ/M4 Session·스트림 위에 얹는 **동기 펌프 드라이버**. Session 계층 수정 금지(완료 핸드셰이크는 앱 레벨).
- 전용 스트림 사용 권장(`session.open_stream()` 또는 인자 `stream_id`). 송수신 양측이 같은 stream_id 합의(예: 기본 전용 stream_id 상수, 또는 OFFER가 흐르는 스트림).
- `FileSender`:
  - `__init__(self, session, clock, *, name: str, data: bytes, stream_id=<전용>, chunk_size: int = 1024)`. 내부 상태(IDLE→OFFERED→SENDING→DONE_SENT→COMPLETE/FAILED), `FileFrameReassembler`.
  - `start(self) -> None`: OFFER(name, size, sha256) 송신(ESTABLISHED 한정).
  - `pump(self) -> 상태/이벤트`: `session.pump()` 구동 + 수신 프레임 처리(ACCEPT 받으면 청크 송신 시작, REJECT면 FAILED; FILE_ACK 받으면 COMPLETE, FILE_NACK면 FAILED) + 송신 진행. **COMPLETE(FILE_ACK 수신) 전에는 close하지 않는다.**
  - `chunk_size`로 data 분할해 CHUNK 프레임들 송신(session.send_on으로; ARQ가 다시 패킷화). 모두 보낸 뒤 DONE 송신.
  - 속성: `state`, `is_complete`, `is_failed`, `progress`(보낸 바이트/전체).
- `FileReceiver`:
  - `__init__(self, session, clock, *, stream_id=<전용>, auto_accept=True)`. 내부: OFFER 수신 전/후 상태, 수신 버퍼(bytearray), 기대 size·sha256.
  - `pump(self) -> 상태/이벤트`: `session.pump()` + 프레임 처리 — OFFER 수신→(auto_accept면)ACCEPT 송신; CHUNK 누적; DONE 수신→재조립 바이트의 SHA-256 == OFFER sha256이면 FILE_ACK + COMPLETE, 아니면 FILE_NACK + FAILED.
  - 속성: `state`, `is_complete`, `is_failed`, `progress`(받은/전체), `file_bytes`(완료 시 원본), `name`, `verified`(bool).
- 완료 핸드셰이크가 flush-on-close를 보장: 송신측은 FILE_ACK(신뢰성 전달) 수신 후에만 종료하므로 모든 청크가 전달·검증된 뒤 닫힌다.
- docstring + 상태 전이 주석.

### M6-T04: 파일 전송 통합 테스트 (deps: M6-T02)

- **변경 파일**: `tests/test_file_transfer.py`
- 무손실·손실 `LoopbackChannel` + `ManualClock` + 두 `Session` 위 `FileSender`/`FileReceiver`:
  - **무손실 전체 전송**: 적당한 크기(예: 수 KB, chunk_size보다 충분히 커 여러 청크) 파일 → 펌프 루프 → 수신측 `is_complete` + `file_bytes == 원본` + `verified` True + SHA 일치. `name` 보존.
  - **손실 채널 전송**: loss>0 + seed 고정 → ARQ 재전송으로 결국 전부 전달·SHA 일치.
  - **완료 핸드셰이크/flush-on-close**: 송신측이 FILE_ACK 수신(COMPLETE) 전에는 종료하지 않음을 확인 — COMPLETE 시점에 수신측이 이미 전체 파일 보유·검증 완료. 송신측 close 후 양쪽 CLOSED.
  - **무결성 실패 경로(선택)**: 의도적으로 손상된 데이터/잘못된 sha를 주면 FILE_NACK + FAILED.
  - **진행률**: 전송 중 progress가 단조 증가, 완료 시 100%.
  - (선택) ImageLoopbackChannel(QR) 위 1건(작은 파일) — segno/cv2 `importorskip`.
  - 결정적(seed/ManualClock), 유한 상한 + 미수렴 시 pytest.fail.

### M6-T05: 파일 전송 예제 (deps: M6-T02)

- **변경 파일**: `examples/file_loopback.py`
- 손실 채널 + 두 Session 위 `FileSender`/`FileReceiver`로 인메모리 "파일"(바이트열) 한 개를 전송 →
  진행률 출력 → 완료 시 SHA-256 일치(MATCH)·수신 크기/이름 출력 → 완료 핸드셰이크 후 양쪽 CLOSED →

## 완료 기준

1. 파일 프레임 코덱이 제어(JSON)·청크(바이너리) 인터리브 프레임을 정확히 라운드트립한다(경계 걸침·전 바이트값 포함), `sha256_hex` 정확.
2. 무손실 채널에서 `FileSender`→`FileReceiver`로 파일을 전송하면 수신 `file_bytes`가 원본과 정확히 일치하고 SHA-256이 검증된다(`verified` True), `name`·크기 보존.
3. 손실 `LoopbackChannel`(seed 고정) + 가상 클럭에서도 ARQ 재전송으로 파일이 전부 전달되고 SHA-256이 일치한다.
4. **완료 핸드셰이크(flush-on-close)**: 송신측은 FILE_DONE 후 FILE_ACK를 받은 뒤에만 종료하며, COMPLETE 시점에 수신측은 전체 파일을 보유·검증한 상태다. 종료 후 양쪽 CLOSED.
5. SHA 불일치(손상) 시 FILE_NACK로 전송이 FAILED 처리된다(무결성 실패 경로).
6. 진행률(progress)이 전송 진행에 따라 단조 증가하고 완료 시 전체에 도달한다.
7. 앱은 Session 계층을 수정하지 않고(순수 추가) 동작하며, 전용 스트림으로 채팅과 병행 가능하다.
8. `examples/file_loopback.py`가 손실 채널에서 파일 전송(SHA MATCH)·완료·양쪽 CLOSED를 출력한다.
9. `pytest`로 M1~M6 테스트가 전부 통과한다(M1~M5 회귀 없음).
