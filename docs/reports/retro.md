# tide 회고 (retro)

> 누적 마일스톤·보고서를 가로질러 본 회고. 회고 시점마다 이 문서 **최상단에** 새 섹션을
> 쌓는다(기존 섹션 보존). `/tide:retro`가 생성·갱신하는 읽기 전용 산출물.

## 2026-06-19 (v0.7.0 시점) — 2차 회고: M1~M7 (파일 전송 + 하드닝까지)

### 집계 범위

- 마일스톤 **M1~M7** (7개), impl 보고서 7개 + review 보고서 7개 (총 21개 문서).
- 릴리즈 **v0.1.0 ~ v0.7.0** (7회, 모두 완료).
- 직전(1차) 회고 이후 추가분: **M6**(파일 전송, v0.6.0) · **M7**(신뢰성 정리/하드닝, v0.7.0).
- 로드맵 0~5단계 + 하드닝 정리 완료. 남은 것: 6단계 실물 광학, 7단계 최적화.

### 반복된 문제·이슈 군집

- **"이월 → 정리 마일스톤" 사이클이 닫혔다** (등장: M3·M4·M5·M6 권장 → M7에서 해소) — 1차 회고가 제안한
  "흩어 넣지 말고 신뢰성 정리 마일스톤으로 묶어라"를 M7로 실행했고, 누적 권장 대부분이 한 번에 해소됨.
  **메타 교훈: 회고의 후속 추적이 실제로 다음 마일스톤 범위를 결정했다**(retro→milestone 피드백 루프가 작동).
- **통합 태스크가 상태머신 결함을 표면화 → 이후 사이클은 결함 0** (등장: M2·M3 결함 / M4·M5·M6·M7 무결함) —
  초기(M2 순차 종료, M3 최종 ACK 손실)엔 통합 태스크가 상태머신 결함을 드러냈으나, **M4 이후 4개 사이클
  연속 소스 결함 0건**. `Channel` 추상 + 순수 컴포넌트 + 동기 펌프 설계가 상위 계층을 무수정으로 얹게 해
  결함 유입을 구조적으로 막은 결과로 보인다.
- **스레드 안전성** (등장: M1·M5·M7) — M1 공유 RNG, M5 cv2 detector로 반복 등장하다 **M7에서 일괄 해소**
  (RNG 락 + 스레드로컬 detector). 단 검증은 "스모크" 수준 — **실제 동시성 정확성은 M6 실물 광학(카메라 스레드)
  에서 본격 검증**으로 이월(처리 추이: 코드 반영 완료, 정확성 검증 이월).
- **단일 프레임 용량/크기 한계** (등장: M3·M5·M6) — NACK 단일 seq, 단일 QR 용량, 파일 chunk_size. M7에서
  QR 용량 가드(`QRCapacityError`)·NACK 억제로 일부 정리. 대용량/스트리밍 전송의 흐름 제어는 여전히 이월.

### 수용된 트레이드오프

- **단일 스레드 동기 펌프 모델** (M2~M7 전반) — 결정적 테스트 우선. M7에서 스레드 안전성 코드는 깔되,
  실 동시성 검증은 실물 광학까지 보류. (1차 회고에서 이어진 일관된 트레이드오프.)
- **acked progress의 프레임 오버헤드 포함** (M7-impl) — `acked_bytes`가 ARQ payload(프레임 헤더 포함)라
  파일 순수 바이트보다 약간 큼 → `min(.,1.0)` 클램프로 처리. 정밀 진행률보다 단순성 선택.
- **데이터 재전송 실패가 폴링 기반** (M7-review) — `SEND_FAILED`를 push 이벤트가 아닌 `data_failed_streams()`
  조회로 노출. 앱은 유한 상한으로 회피 중. push화는 후속.
- **파일 단방향·단일 파일, half-close 미지원** (M6·M2~) — 완료 기준 충족 우선, 양방향/다중/재개는 이월.
- **base64 33% 용량 증가** (M5) — ASCII-safe 바이너리 무결성과 맞바꿈(유지).

### 릴리즈 판정·버전 추이

- **M1**: 가능 → v0.1.0 (최초, 범프 없음) / **M2**: 가능 → v0.2.0 / **M3**: 가능 → v0.3.0 / **M4**: 가능 → v0.4.0
  / **M5**: 가능 → v0.5.0 / **M6**: 가능 → v0.6.0 (파일 전송) / **M7**: 가능 → v0.7.0 (하드닝).

  M4~M7은 **impl 단계 소스 결함 0건**(M2·M3만 통합 결함을 impl에서 수정).

### 회고 메모

1. **최우선 미반영 후속 = 스레드 동시성 정확성 검증**, 그리고 그것이 곧 **6단계 실물 광학** 마일스톤의
   일부다. M7이 스레드 안전성 코드를 깔아두었으니, 실물 광학에서 카메라 백그라운드 스레드를 도입하며
   동시 송수신 정확성을 함께 검증하는 것이 자연스럽다.
2. **실물 광학은 결정적 자동 테스트가 어렵다**(하드웨어·대화형). 마일스톤 설계 시 "타이밍·프레임 버퍼링
   로직(자동화 가능)"과 "실제 화면·카메라 수동 검증 절차"를 분리해 완료 기준을 세울 것.
3. **남은 미반영은 대부분 저우선/기능 확장**(arq max_retx 노출·실패 push·latency 채널·half-close·다중 파일·
   FEC). 다음 정리 마일스톤이 필요할 만큼 쌓이기 전까지는 기능(실물 광학) 진행이 합리적.
4. **검증된 패턴 유지**: ① 손실/통합 테스트 표준화, ② `Channel` 추상 위 순수 컴포넌트 + 동기 펌프(결함 유입
   차단), ③ 사전 확정 설계 질문(매 마일스톤 전), ④ 회고→정리 마일스톤 피드백 루프.
5. **1.0 후보**: 실물 광학(M6 단계)까지 완료해 "빛만으로 채팅·파일 실통신"이 실증되면 v1.0 선언 시점.

## 2026-06-19 (v0.5.0 시점) — 첫 회고: M1~M5(기본 통신 + QR 코덱 실물 통합)까지

### 집계 범위

- 마일스톤 **M1~M5** (5개), impl 보고서 5개 + review 보고서 5개 (총 15개 문서).
- 릴리즈 **v0.1.0 ~ v0.5.0** (5회, 모두 릴리즈 완료).
- 로드맵 0~4단계 완료(패킷 → 세션 → 신뢰성 ARQ → 스트림+채팅(기본 통신) → QR 코덱 실물).

### 반복된 문제·이슈 군집

- **단일 스레드 모델의 스레드 안전성** (등장: M1, M5) — M1 `LoopbackChannel` 공유 RNG, M5 cv2 `QRCodeDetector` 공유 인스턴스가 스레드 안전이 아님. 처리 추이: **의식적 이월** — 전 계층이 "I/O 비의존 순수 컴포넌트 + 동기 펌프(단일 스레드)" 모델이라 현재는 무의미. **M6 실물 광학(카메라 백그라운드 스레드) 도입 시 일괄 해소 필요.**
- **통합 태스크가 상태머신 결함을 표면화** (등장: M2, M3) — M2 `T04`(드라이버 통합)가 순차 종료 시 수동측 `CLOSE_WAIT` 잔류를 발견 → 대칭 auto-close로 수정. M3 `T08`(손실 통합 테스트)가 핸드셰이크 최종 ACK 손실 복구 불가를 발견 → ESTABLISHED에서 재전송 SYN_ACK에 ACK 재전송으로 수정. 처리 추이: **둘 다 impl 단계에서 메인이 수정 + 결정적 회귀 테스트 추가**(seed 운에 기대지 않음). 패턴: 손실/통합 시나리오 테스트가 상위 결함을 조기에 드러낸다.
- **타임아웃/재전송 견고성** (등장: M1, M2, M3) — M1 latency/jitter 미구현, M2 수립 타임아웃 부재, M3에서 제어 패킷 재전송 + 수립 타임아웃 + 최종 ACK 복구로 **대거 해소**. 처리 추이: M3가 누적 권장을 흡수.
- **패키지 `__init__` re-export 일관성** (등장: M1, M4, M5) — M1 review에서 packet/channel re-export 누락을 수정·규약화한 뒤, M4(Session 통합)·M5(`ImageLoopbackChannel` 추가)가 그 규약을 준수. 처리 추이: **수정 후 후속 준수**(정착).
- **단일 프레임 용량/크기 한계** (등장: M3, M5) — M3 NACK가 최저 누락 seq 1개만 표현, M5 단일 QR 심볼 용량 한계·소형 QR 검출 하한. 처리 추이: **이월**(현 기본값에서 안전, 파일 전송/대용량 시 재설계).
- **문서-실동작 일치** (등장: M1, M2) — M1 docstring 한/영 혼재(사소), M2 종료 절 산문 vs 대칭 auto-close 구현 불일치를 review에서 정정. 처리 추이: M2 수정, M1 사소 잔존.

### 수용된 트레이드오프

- **단일 스레드 동기 펌프 모델** (출처: M2~M5 전반) — 결정적·재현 가능 테스트를 위해 백그라운드 스레드 없이 명시적 `pump()`. 대가로 스레드 안전성/동시성은 비범위(M6까지 보류).
- **가상 클럭 기반 RTT/RTO 검증, 채널 전달 지연 미구현** (출처: M1 사소 → M3 의식적 이월) — `ManualClock` 전진으로 RTO·타임아웃을 결정적으로 검증. `LoopbackChannel`의 실제 전달 지연(latency/jitter)은 동기·클럭 비주입 큐 구조상 7단계로 이월.
- **base64 ASCII-safe 래핑(용량 ~33% 증가)** (출처: M5-impl) — QR 디코더의 문자열 반환에서 바이너리 손상을 막기 위함. throughput 비우선 프로젝트라 수용.
- **대칭 auto-close, 반이중(half-close) 미지원** (출처: M2~M4) — "단일 close()로 양쪽 CLOSED" 완료 기준 충족을 우선. per-stream 수명주기는 파일 전송에서 필요 시 확장.
- **NACK 단일 누락 seq** (출처: M3-impl) — 단순화. 구멍이 여러 개여도 라운드당 1개씩 결국 복구.
- **종료 시 FIN_ACK 손실 → `TIMED_OUT` 경로로 CLOSED** (출처: M3-impl, M4-review) — 양쪽 CLOSED 도달은 보장하되 완료 이벤트가 `CLOSED`가 아닐 수 있음(이벤트 정합성은 후속).

- **M1**: 가능 → **v0.1.0** (최초 릴리즈, 등록 이력 없어 초기 스캐폴드 버전 그대로 — 범프 없음). 패킷/채널 토대.
- **M2**: 가능 → **v0.2.0** (minor). 세션 계층. impl 중 종료 결함 수정.
- **M3**: 가능 → **v0.3.0** (minor). 신뢰성 ARQ. impl 중 최종-ACK-손실 복구 수정, M1·M2 권장 해소.
- **M4**: 가능 → **v0.4.0** (minor). 스트림+채팅 = 기본 통신 완성. 소스 결함 0건.
- **M5**: 가능 → **v0.5.0** (minor). QR 코덱 실물 통합. 소스 결함 0건, 첫 외부 의존성 도입.

### 회고 메모

1. **최우선 미반영 후속 = flush-on-close (M4-review).** 파일 전송 마일스톤에서 "전송 완료 후 종료" 보장이 없으면 마지막 청크 유실 위험. 파일 전송 착수 시 **필수 반영**.
2. **M3 리뷰 권장 4건이 계속 이월 중**(NACK 억제·데이터 idle 타이머·재전송 상한/파라미터 노출·종료 이벤트 정합성). 다중 스트림·장기 전송에서 영향이 누적될 수 있어, 흩어 넣기보다 **"신뢰성 정리(hardening) 마일스톤"으로 묶어** 처리하는 편이 효율적.
3. **스레드 안전성(공유 RNG·cv2 detector)은 M6 실물 광학에서 카메라 스레드를 도입하는 시점에 일괄 해소**. 그 전엔 단일 스레드 모델이라 비용 대비 이득 없음 — 지금 손대지 말 것.
4. **통합/손실 시나리오 테스트를 일찍 두는 것이 효과적**이었다(M2·M3 상태머신 결함을 통합 태스크가 조기 발견). 새 마일스톤마다 "손실 채널 + 통합" 테스트를 표준으로 유지.
5. **단일 프레임 용량(QR) ↔ 파일 전송 청크 크기 설계가 맞물린다.** 파일 전송 마일스톤에서 `max_payload` 상한(QR 용량)과 청크/분할 전략을 함께 설계(M5 권장 1과 연계).
6. **버전 규율 정착**: 최초 릴리즈는 초기 버전 그대로, 이후 신규 계층마다 minor. 1.0 진입은 실물 광학(M6)·파일 전송까지 완료해 "절대적 매체 위 실 통신"이 증명된 시점이 자연스러운 후보.
