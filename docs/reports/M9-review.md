# M9 리뷰보고서 (review)

## 비판점

### 차단 (0건)

하드닝 릴리즈를 막는 차단 이슈 없음. CI로 검증 가능한 완료 기준(1·2·3·4·7·9)이 충족되고, 전체
**175 테스트 통과**(M1~M8 172 + M9 신규 3, 회귀 0), 예제 `--help`·인자 가드 정상. 페이싱·버퍼 드레인·
파라미터 가드·윈도우 dedup 구현이 정확하고 기존 패턴과 일관된다. **단, 완료 기준 5·6·8의 "실 하드웨어
동작"과 v1.0 게이트는 CI에서 검증 불가**(권장 3 참조) — 이는 마일스톤이 설계상 수동 사인오프로 갈음하기로
한 부분이다.

검증한 핵심 정확성:
- **윈도우 dedup**: `_capture_loop`이 `deque(maxlen=8)` + 미러 `set`으로 재캡처를 O(1) 판정. 퇴거 시
  `recent_set.discard(recent[0])` 후 append로 두 뷰 동기 유지(정확). 순서 흔들림(A,B,A,C)에서 두 번째 A를
  떨구고 A·B·C를 각 1회 순서대로 인도 — 신규 테스트로 결정적 검증. 단일 슬롯이라면 오인도하는 케이스.
- **페이싱**: `send_frame`이 `hold>0`에서만 단조 시계로 부족분을 sleep, `hold==0`은 타임스탬프·sleep 없이
  단락(기존 동작 불변 — 차등 테스트로 확인). nonce는 show 성공 후에만 전진(QRCapacityError에 nonce 미소모).
- **가드**: `poll_interval<=0` → `_MIN_POLL_INTERVAL=1e-3` 클램프, `hold<0` → 0. `MemoryCamera`도
  repeat_last+falsy timeout 시 `_MIN_REPEAT_WAIT` 하한(채널 경로 무영향).
- **Cv2Camera 드레인**: `_MAX_DRAIN=4` 고정 한도 grab/retrieve로 최신 프레임 취득(무한 루프·과지연 없음),
  `CAP_PROP_BUFFERSIZE=1`·해상도 best-effort guarded, import 하드웨어 비접촉.

### 권장 (3건)

1. **완료 기준 4의 "다중 바이트 nonce" 미충족 — 윈도우 dedup으로 대체(설계 변경, 수용 판정 필요)** —
   마일스톤은 다중 바이트 nonce를 명시했으나, 구현은 **nonce 1바이트 유지 + 윈도우 dedup**으로 갔다.
   사유는 타당하다(2바이트가 한 세션 프레임을 cv2 디코드 블라인드스폿에 빠뜨려 회귀 유발, 업스케일
   폴백도 무효 — impl 단계에서 실측). 다만 impl 보고서의 *"윈도우만으로 래핑 거짓 dedup이 이미 제거"*는
   **약간 과장**이다: 윈도우(8) ≪ 래핑 주기(256)는 거짓 dedup을 *매우 어렵게* 만들지만 수학적으로
   완전 제거하진 않는다 — 극단적 캡처 손실(>96%)로 8개 인도분이 256 send를 가로지르고 그 안에 동일
   `channel_frame`이 재등장하면 이론상 거짓 dedup이 가능하다. **실 제거는 윈도우가 아니라 프로토콜
   특성**(M7의 `max_retx=8` 재전송 상한 + ARQ seq로 패킷이 256-send 범위에서 사실상 유일)에 기인한다 —
   동일 `channel_frame`이 256 send 간격으로 재등장할 수 없으므로 실 시스템에선 발생 불가. 따라서 criterion
   4의 *의도*(거짓 dedup 없음)는 실질 충족되나 *문구*(다중 바이트 nonce)는 미충족. 트레이드오프는
   합리적이되, v1.0 실 링크 견고성을 위해 **nonce 폭을 QR 콘텐츠에서 분리**(디코드 견고화 후 폭 확대,
   또는 nonce를 패킷 코덱이 아닌 별도 채널 헤더에 배치)하는 보강을 후속 검토할 것.
2. **cv2 디코드 콘텐츠 의존 블라인드스폿 (미해결, 별도 후속)** — 특정 v8 QR을 cv2 `QRCodeDetector`가
   어느 배율로도 디코드 못 하는 사례가 확인됐다. 실 링크에선 ARQ가 새 nonce→새 이미지로 재전송해
   회복하지만 **수신율을 깎고 지연을 늘린다**(셀프체크 하니스가 정확히 측정할 항목). 대체 detector·전처리
   (이진화/샤프닝)·EC 레벨 상향 등 **디코드 견고화**는 실 수신율과 v1.0 통과율에 직결되는 후속 과제.
3. **v1.0 게이트(실 하드웨어 사인오프) 미수행 → 이번 릴리즈는 하드닝(v0.9.0)** — 셀프체크·2-머신
   하니스와 절차·PASS 기준(수신율 ≥80% + 2-머신 ESTABLISHED/MATCH/CLOSED)은 갖춰졌으나, 실제 카메라
   검증 결과가 어디에도 기록되지 않았다(CI 카메라 부재). 마일스톤이 명시한 폴백대로 **하드닝만 v0.9.0
   (minor)으로 릴리즈**하고, 실 하드웨어 사인오프를 통과한 뒤 v1.0으로 승격해야 한다(아래 판정).

### 사소 (2건)

4. **`--real`(role 없이)은 여전히 디스플레이만** — M8과 동일(인메모리 루프 + 실 화면 표시). 실 왕복은
   `--real --role sender|receiver` 2-머신 모드가 담당하며, 문서·help가 이를 일관되게 안내한다(정상).
   `--camera` 인자는 role 모드에서만 의미. 혼동 방지 문구 유지 양호.
5. **`_MAX_DRAIN=4`·`CAP_PROP_BUFFERSIZE` best-effort의 백엔드 의존성** — 드라이버/OS별 효과가 달라 실
   지연 감소 폭이 환경마다 다르다(드레인이 실질 안전망). 셀프체크로 환경별 `--hold`/`--scale`과 함께
   실측 튜닝 권장(문서화돼 있음).

  발생 불가로 판단하나, 실 극단 손실 환경에서의 거동은 미측정.
