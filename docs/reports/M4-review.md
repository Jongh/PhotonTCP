# M4 리뷰보고서 (review)

## 비판점

### 차단 (0건)

차단 이슈 없음. 완료 기준 1~8 충족, 전체 테스트 95건 통과, 예제 4종 정상. 이번 사이클은 서브에이전트가
소스 결함을 발견하지 않았다(M3의 최종-ACK-손실 수정 덕에 손실 채팅 시나리오가 seed 제약 없이 수렴).

### 권장 (3건)

1. **세션 종료 시 스트림 데이터 flush 보장 없음** — `close()`는 in-flight 스트림 데이터의 완전 전달을
   기다리지 않는다(graceful 종료와 데이터 완료의 순서 보장 없음). 채팅에서 "메시지 send 직후 곧장 close"하면
   미전달분이 유실될 수 있다. 채팅 데모/테스트는 close 전에 충분히 펌프해 회피 중. **파일 전송(M5)에서는
   "전송 완료 후 종료" 보장이 필수**이므로 거기서 flush-on-close 또는 per-stream FIN을 도입할 것.
2. **`ChatSession.received`가 내부 가변 리스트를 그대로 노출** — 호출자가 실수로 변형 가능(docstring은
   read-only 안내). 방어적으로 복사본 반환 또는 튜플화 권장(저위험).
3. **Session `rto` 인자의 factory 변환이 `RtoEstimator` 비공개 속성에 약결합** — 전달된 추정기의
   `initial/min/max`를 비공개 속성에서 읽어 스트림별 복제. `getattr` 기본값 방어는 있으나, `RtoEstimator`에
   공개 설정 접근자 또는 `clone()`을 추가하면 결합이 깔끔해진다.

### 사소 (2건)

4. **M3 리뷰 권장 4건 여전히 이월(M4 비범위 확정)** — NACK 억제·데이터 활동 idle 타이머·데이터 재전송
   상한/파라미터 노출·종료 이벤트 정합성. 다중 스트림에서 NACK 스톰·idle 결합 영향이 커질 수 있어 **정리
   마일스톤**으로 묶어 처리 권장.
5. **반이중(half-close)·per-stream 수명주기 미지원** — 스트림 종료는 세션 종료와 함께(암묵적 경량 설계대로).
   파일 전송 다중화에서 스트림 단위 종료가 필요해지면 확장.

## 수정 내용

- 차단 0건이라 리뷰 단계의 소스 수정은 없음.

## 검증

- `python -m pytest -q` → **95 passed in 2.67s** (M1 13 + M2 17 + M3 42 + M4 23, M1~M3 회귀 없음).
- 예제 4종: `echo_loopback`·`session_loopback`·`reliable_loopback`·`chat_loopback` 전부 정상
  (채팅: loss 20% 링크에서 양방향 4+4 메시지 순서대로 MATCH, 양쪽 CLOSED).
- 스트림 격리(HOL 없음)·implicit open·패리티·코덱 재조립·유니코드·손실 양방향 채팅·레거시 호환 모두 테스트로 커버.
- 잔여 리스크: 권장 1(flush-on-close)은 M5 파일 전송에서 반드시 다뤄야 함. 권장 4(M3 이월)는 다중 스트림에서
  효율 영향이 누적될 수 있음. 현재 완료 기준(결정적 양방향 채팅)에서는 문제 없음.
