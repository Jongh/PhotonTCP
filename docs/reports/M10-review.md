# M10 리뷰보고서 (review)

## 비판점

### 차단 (0건)

릴리즈를 막는 차단 이슈 없음. 마일스톤 완료 기준 1~7이 모두 충족되고(아래 검증), 전체
**188 테스트 통과**(M9 175 + M10 신규 13, 회귀 0), 벤치 스크립트가 디코드율 개선을 정량
입증한다(깨끗 base 96.14%→풀 **100%**, 열화 97.10%→풀 **100%**). 검증한 핵심 정확성:

- **전처리 캐스케이드**: `_decode_variants`가 변형을 지연 생성(generator)하고 `decode_frame`이
  첫 성공에서 단락 — 깨끗한 QR은 변형 1에서 즉시 성공해 변형 2~4를 생성조차 안 한다(핫패스
  비용 0, 단위 테스트로 detectAndDecode 정확히 1회 호출 확인). `cv2.error`·잘못된 base64를
  **변형 단위**로 흡수하므로 한 변형의 실패가 전체를 죽이지 않는다. None 계약·never-raise 불변.
- **대체 detector 폴백**: 1차가 전 변형 실패 시에만 진입(콜드패스), 가용성 1회 probe 후 캐시,
  인스턴스 thread-local(1차 detector와 동일 패턴 — 캡처 스레드 안전). 미가용 환경은 폴백 생략으로
  캐스케이드 단독과 동일(우아한 degrade). 폴백 부재/회복 양 경로를 단위 테스트로 결정적 검증.
- **EC 배선**: `error`가 `pair`→`__init__`→`send_frame`→`encode_frame`로 관통, `"q"` 왕복·기본
  `"m"` 호환·`"q"`≠`"m"` 인코딩 차이를 테스트로 확인.
- **nonce 2바이트 확대**: 1B/2B decode-rate 스윕 양쪽 100% + 전 suite 그린으로 재검증 후 확대,
  2바이트 프레이밍·증가를 회귀 테스트로 핀. M9가 보류한 "다중 바이트 nonce"(완료 기준 4 문구)를
  디코드 견고화 덕에 실제로 달성.

### 권장 (2건)

1. **Micro-QR 블라인드스폿은 cv2 한계가 아니라 인코더가 원천 — `encode_frame(micro=False)` 한
   줄로 제거 가능(후속 권장, 리뷰에서 미수정)** — impl/코퍼스/`test_micro_qr_boundary_excluded`는
   "≤9바이트 페이로드는 cv2가 어느 배율로도 디코드 못 하는 직교 한계"로 문서화했으나, 리뷰 중
   실측 결과 그 원인은 **인코더**에 있다: `segno.make(b64)`가 짧은 데이터에 기본 `micro=True`로
   **Micro QR**(M3/M4)을 만들고, cv2 `QRCodeDetector`는 Micro QR을 지원하지 않는다. 같은 데이터를
   `segno.make(b64, micro=False)`로 인코딩하면 5·8·9바이트 페이로드도 **version 1 일반 QR로 100%
   디코드**된다(리뷰에서 확인). 즉 "디코드 견고화"의 가장 근본적 한 수는 디코더가 아니라 **인코더가
   cv2-디코드 불가 심볼을 애초에 내보내지 않게** 하는 것이다.
   - **비차단 사유**: 본 프로토콜 패킷은 최소 22바이트 헤더(+2바이트 nonce) → base64 ~32자 →
     항상 일반 QR이라 실사용에서 Micro 경계에 닿지 않는다. 그래서 릴리즈를 막지 않는다.
   - **리뷰에서 안 고친 사유(스코프)**: M10은 `encode.py`를 의도적으로 범위 밖에 뒀고,
     `micro=False`를 넣으면 `test_micro_qr_boundary_excluded`(8바이트가 *실패*함을 단언)가 깨지며
     코퍼스 하한(≥12B)도 무의미해져 **연동 변경**이 필요하다 — 리뷰 핫픽스보다 작은 후속 태스크가
     적합. 다음 사이클 또는 v0.10.x 패치에서 `encode_frame`에 `micro=False`(또는 인자화) + 경계
     테스트/코퍼스 조정으로 블라인드스폿을 *원천 제거*할 것을 강력 권장.

2. **대체 detector의 환경 의존성 — 실 배포 가이드 필요** — 현재 환경은 cv2 4.13에서 `aruco`가
   가용해 폴백이 활성이나, **코어 opencv-python**(WeChat·Aruco 모두 부재) 환경에서는 폴백이
   생략되어 캐스케이드 단독으로 동작한다(회복폭 축소). 실 광학 링크 배포 시 opencv-contrib
   (WeChat) 설치를 권장하고, 셀프체크/2-머신 실측 때 **어떤 대체 detector가 활성인지**를 함께
   기록해야 수신율 수치의 재현·비교가 가능하다(벤치는 이미 활성 detector를 출력하므로 절차에
   반영만 하면 됨).

### 사소 (2건)

3. **캐스케이드의 이론적 base64 거짓-성공** — `decode_frame`은 첫 *base64 검증 통과* 결과를
   반환하므로, 어떤 변형이 다른 QR을 오검출하고 그 텍스트가 우연히 base64로 디코드되면 잘못된
   바이트를 반환할 수 있다. 실제로는 QR의 자체 EC+포맷 정보가 오검출을 사실상 차단하고, Otsu는
   이미 이진인 콘텐츠를 거의 그대로 두며, 패킷에는 상위 계층 CRC가 있어 무해하다. 또한 M10 이전
   단일패스도 동일한 이론적 노출이 있었다(신규 리스크 아님). 기록만.

4. **`_alt_decode`가 `_decode_variants`를 재실행** — 폴백 진입 시 Otsu/샤프닝/업스케일이 재계산된다.
   1차가 전 변형 실패한 콜드패스에서만 발생하므로 비용상 무해(핫패스 불변). 향후 폴백이 잦은
   환경이 생기면 변형 캐싱을 검토.

## 수정 내용

- **차단 0건 → 리뷰 단계의 소스 수정 없음.** 권장 1(인코더 `micro=False`)은 비차단이며
  `encode.py`(M10 범위 밖) + 경계 테스트/코퍼스의 연동 변경을 수반해 리뷰 핫픽스로 처리하지
  않고 후속 태스크로 넘긴다(위 사유). impl 단계에서 이미 nonce 2바이트 재검증 + 페이싱 테스트
  플레이키(차등 타이밍 → 단측 경계 2개)를 해소했고, 그 결정들은 위 차단/검증에서 평가했다.

## 검증

- `python -m pytest -q` → **188 passed**(리뷰 재실행, ~18.9s; impl과 동일). 신규 13 = test_qr 5 +
  test_qr_robustness 4 + test_optical_ec 3 + test_optical_pacing nonce 1.
- `python examples/qr_decode_bench.py` → exit 0. 깨끗 base 96.14%→풀 100%(+8), 열화 97.10%→
  풀 100%(+6), "full ≥ base-only on every set: YES", 활성 대체 detector `'aruco'` 출력.
- 환경 실측: cv2 4.13.0, `_resolve_alt_kind()`=`"aruco"`(WeChat attr 부재). `micro=False` 시
  5/8/9바이트 페이로드도 version 1로 100% 디코드(권장 1 근거).
- 완료 기준 매핑: 1(핫패스 불변·단위테스트)·2(블라인드스폿 회복, 벤치 +8/+6 & 회복 테스트)·
  3(대체 미가용 None·never-raise 테스트)·4(`error="q"` 왕복 테스트)·5(벤치 exit 0, on≥off)·
  6(suite 188 + 코퍼스 임계 통과)·7(nonce 재검증 결과·근거 impl 기록) — **전부 충족**.
- **잔여 리스크(솔직히 명시)**: (a) 실 카메라 수신율·2-머신 왕복은 하드웨어 부재로 이번에도
  미검증 — v1.0 승격 전 수동 사인오프 필수(M9와 동일). (b) 코어 opencv-python 환경의 회복폭은
  본 환경(aruco 활성)과 다를 수 있어 미측정(권장 2). (c) 열화 코퍼스는 의도적으로 가벼워(풀
  100%) 임계 0.85에 여유가 크다 — 강한 실 열화는 셀프체크로만 실측 가능.

## 릴리즈 판정

**가능** — 추천 버전: **v0.10.0 (minor)**

- 완료 기준 1~7 충족, 차단 0건. 변경은 모두 **backward-compatible 기능 추가/견고화**: `decode_frame`은
  공개 시그니처·None 계약 불변이나 회복력이 올랐고, `OpticalChannel`의 `error` 인자는 기존부터
  존재(문서·테스트 보강), 신규 산출물(`qr_decode_bench.py`)·테스트가 추가됐다. 공개 API 파괴 없음 →
  **minor**. 기반 v0.9.0 → **v0.10.0**.
- **v1.0.0(major) 아님**: 마일스톤·M9가 규정한 v1.0 게이트는 "빛만으로 실 통신"을 실 하드웨어에서
  증명하는 **수동 사인오프**(셀프체크 ≥80% + 2-머신 ESTABLISHED/MATCH/CLOSED)를 전제로 한다.
  M10은 그 게이트의 **수신율 상한을 끌어올렸을 뿐** 사인오프를 수행하지 않았으므로 v1.0을 주장할
  수 없다. 권장 1·2는 후속(또는 v1.0 절차)이 집어갈 항목으로 하드닝 릴리즈를 막지 않는다.
